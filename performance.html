Performance notes

There could be several ways to do stuff. Which one is faster? See comparison of single value assignment in data.frame, data.table and list. It seems that it’s faster to read from higher level construct but write to lower level construct.

Is magrittr’s two-way pipe syntax good?

Magrittr two way pipe syntax looks convenient, as one doesn’t need to write a variable name twice if something is applied to it. But I experienced it to slow a cycle significantly in a project of mine. Is it slow indeed? What about one-way pipe operator, is that one slow as well? Is it different in dplyr, if dplyr imports magrittr’s namespace?

Seems that pipe is at least 10 times slower than straight assignment (and dplyr just reexports magrittr’s operator) - and so it can be used between functions that handle frames, but not inside of cycles.

Strsplit is faster than regexp

cleanup.strsplit.NA <- function(s) {
        # This one is broken - it will return NA if the split isn't found
        strsplit(x=s, split="oz.) ", fixed=TRUE)[[1]][2]
}

cleanup.strsplit <- function(s) {
        s1 <- strsplit(x=s, split="oz.) ", fixed=TRUE)[[1]][2]
        if(is.na(s1)) s else s1
}

cleanup.sub <- function(s) {
        sub("(.+oz\\.\\) )(.+)", "\\2", s)
}

l <- rep(c("simple","(   oz.) tomato paste"),10)
microbenchmark(
        lapply(l,cleanup.sub),
        lapply(l,cleanup.strsplit),
        lapply(l,cleanup.strsplit.NA),
        times=1000L
        ) %>%
        autoplot

Apply vs for loop

Technically, apply function do the same thing as for loops, but if something can be organized to be used in apply(), then it’s probably more efficient as well.

Code below calculates cuisine per receipt based on ingridient list. Probability of cuisine for each ingredient is in “weights”. First code in two variations (with pipes and not) ran for 44 and 41 seconds on list of 10k receipts, while second code (assigned to matrix from a list-based cycle) ran for 83 seconds.

Interesingly, the code below shows opposite dynamic - list is faster. A possible explanation is that apply() works better when handling large objects thousands of rows - then list-style assignment of separate values becomes the bottleneck, while on short test lists setting up higher level construct of apply() takes more time. So, again higher level construct performs better on more complex data, where effort spent on optimization and preparation is relatively small compared to number of following repetitions.

probs.cut <- weights
CuisineByIngr.pipes <- function(ings) {
        cuisines[
                lapply(ings, function(y) probs.cut[y,]) %>%
                unlist %>%
                matrix(ncol=20, byrow=TRUE) %>%
                apply(2,sum) %>%
                which.max
                ]
}

CuisineByIngr.nopipes <- function(ings) {
        cuisines[
                which.max(
                        apply(
                                matrix(
                                        unlist(
                                                lapply(ings,
                                                       function(y)
                                                               probs.cut[y,]
                                                       )
                                                ),
                                        ncol=20,
                                        byrow=TRUE),
                                2,
                                sum)
                )
                ]
}
